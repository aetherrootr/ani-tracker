# Ani Tracker 设计风格

## 文档权威性

本文档是 Ani Tracker 设计语言的中文参考译本。

- 英文文档 `docs/design_style.md` 是唯一规范源。
- 本文档 `docs/design_style.zh-CN.md` 仅用于中文阅读和讨论。
- 如果中英文文档存在差异，以英文文档为准。
- 产品与实现决策必须持续参考 Apple Human Interface Guidelines（HIG），并将其原则适配到 Web 和各平台环境。
- 桌面端与移动端是同等重要的产品目标，任何一端都不是另一端的简化版本。

## 产品背景

Ani Tracker 是一个跨平台动画追踪应用。它必须支持用户选择二次元插画作为整个网页最底层的背景。

这些壁纸可能具有：

- 高饱和度的幻想色彩；
- 密集的人物、服饰与装饰细节；
- 同一画面中并存的明亮高光与深色阴影；
- 强烈的互补色对比；
- 类似文字的图形、线稿、粒子和装饰纹样；
- 接近当代高密度、高色彩表现力二次元插画的视觉复杂度。

这是不可妥协的环境约束。毛玻璃语言不是为了追随视觉潮流而添加的装饰，而是让导航、控件和内容在保留用户喜爱插画的同时仍然清晰可读的功能层。

设计系统必须适应未知壁纸，而不能只在默认 Twilight Iris 渐变背景上成立。

## 设计名称

**Ani Tracker - Twilight Iris（暮色鸢尾）**

Twilight Iris 由以下元素构成：

- Iris Indigo（鸢尾靛紫）：选择、焦点和主要操作；
- Periwinkle（长春花紫）：环境光与氛围连续性；
- Emerald（祖母绿）：已观看、完成和成功；
- Neutral Graphite（中性石墨）：内容与结构；
- 自适应玻璃与中性遮罩：确保高表现力插画背景上的可读性。

产品应当呈现平静、精确、尊重媒体内容且具有个性化的感受。它应尊重鲜艳的动画插画，但不能让插画压过用户当前的任务。

## Apple 设计原则

Apple HIG 是 Ani Tracker 长期且固定的设计输入。我们不会机械复制 Apple 界面，而是将其原则转化为响应式 Web 行为。

### 层级

- 按阅读顺序优先展示主要内容和操作。
- 将相关控件组合在一起，将无关功能分隔开。
- 保持内容层与导航、控制层之间的清晰区别。
- 对次要信息和低频维护工具使用渐进披露。

### 可读性

- 所有文字和控件都必须在每种受支持壁纸上保持可读。
- 不能只依赖模糊。模糊必须与足够的不透明度、明度控制和对比度结合。
- 使用语义化的前景层级：主要、次要、三级。
- 小字号文字避免使用纤细字重。

### 适应性

- 桌面端和移动端可以使用不同布局、导航、组件与操作逻辑。
- 跨平台保持品牌、语义、任务结果和反馈一致。
- 尊重安全区域、浏览器界面、虚拟键盘、可调整窗口、本地化和文字缩放。

### 熟悉的交互

- 优先使用用户熟悉、符合平台习惯的控件和呈现方式。
- 桌面端可使用 Sidebar、锚定 Popover、Inspector、Dialog 和侧边 Sheet。
- 移动端使用顶部导航作为 Ani Tracker 的标志性模式，并在适合时结合 Bottom Sheet、全屏流程和直接手势。
- Hover 只能是增强，不能成为唯一操作入口。
- 每个手势都必须存在可见且可访问的替代操作。

### 包容性设计

- 不依赖颜色单独表达关键状态。
- 支持键盘操作和清晰可见的焦点。
- 尊重减少动态效果和减少透明度偏好。
- 保持足够的交互热区与对比度。

## 基础层级模型

界面由五个概念层构成。即使平台布局发生变化，它们的顺序也保持稳定。

### Layer 0 - 用户插画

用户壁纸是最底层视觉层。

职责：

- 表达个性化；
- 营造氛围；
- 保持足够可见，使个性化有意义；
- 永远不承担文字或控件对比度责任。

壁纸是类似内容的艺术图像，不是 UI 色彩 token。不得从某张壁纸随意提取一种颜色并直接应用到全部控件。

### Layer 1 - 自适应环境层

该层在 UI 覆盖前稳定壁纸表现。

它可以包括：

- 中性亮色或暗色遮罩；
- 低强度 Twilight Iris 色调；
- 导航背后的局部明度抑制；
- 可选的背景模糊或柔化；
- 保护安全区域和 Sticky 控件的边缘渐变。

环境层必须减少视觉噪声，但不能把壁纸变成无法辨认的灰雾。

### Layer 2 - 内容表面

该层包含动画卡片、剧集行、元数据、正文、图表与设置分组。

规则：

- 优先使用实体或高不透明度表面。
- 使用克制的边框与低层级阴影。
- 不要为长列表中的每张卡片添加重度背景模糊。
- 卡片中的海报与文字区域应保持视觉分离。
- 元数据表面与插画光晕重叠时必须使用中性遮罩。

### Layer 3 - 功能玻璃层

该层包含悬浮在内容上方的导航与控件。

示例：

- 桌面 Sidebar；
- 移动顶部导航；
- Sticky 搜索与工具栏；
- Segmented Control 轨道；
- 锚定 Popover；
- 悬浮操作控件。

功能玻璃应明确表现为高于内容的一个平面。

### Layer 4 - 模态与聚焦表面

该层包含 Dialog、Sheet、壁纸上方菜单、数据源切换和长文本阅读层。

规则：

- 使用高密度、接近实体的玻璃。
- 使用背景遮罩将聚焦任务与页面分离。
- 可能发生裁切或层级不稳定时，通过 Portal 渲染。
- 保留明确的关闭操作，并在适合的平台支持 Escape。

## 壁纸适配契约

每个页面都必须在明亮、深色、高饱和、低对比和构图复杂的壁纸上可用。

### 壁纸渲染

- 保持插画宽高比。
- 应用背景优先使用 `cover`；如果支持焦点位置，应尊重用户设置。
- 不拉伸图像。
- 避免在导航和安全区域周围产生突兀裁色。
- 除非动态效果具有明确产品目的，否则壁纸应保持固定或空间稳定。
- 默认不使用壁纸缩放动画或视差。

### 可读性保护

壁纸与文字密集 UI 之间必须至少存在一层中性保护。

推荐顺序：

```text
用户插画
  -> 自适应遮罩 / 色调
  -> 内容或功能表面
  -> 文字与控件
```

正文不得直接放在未经处理的壁纸上。

### 明暗区域并存

一张壁纸可能同时存在明亮和深色区域，因此：

- 仅依赖全局亮色/深色模式不够；
- 大型玻璃表面需要足够自身不透明度，以同时应对两个极端；
- 导航表面可以使用局部遮罩；
- 海报或 Hero 光晕必须在进入元数据和控件区域前衰减；
- 玻璃后方的白色高光不能让边框、图标或次要文字消失。

### 用户壁纸控制

如果提供壁纸控制，应优先提供少量、易理解的设置：

- 选择图像；
- 焦点位置；
- 压暗强度；
- 模糊/柔化强度；
- 恢复默认背景。

不要用任意主题色自定义代替可靠的壁纸适配。Twilight Iris 的语义色板应在所有壁纸上保持一致。

### 降级方案

- 不支持 `backdrop-filter` 时，提高表面不透明度并保持层级。
- 在 `prefers-reduced-transparency` 下使用实体表面。
- 图片加载失败时使用默认 Twilight Iris Canvas 和环境渐变。
- 壁纸加载不得造成明显布局跳动。

## 材质系统

玻璃材质按功能和阅读负荷选择，而不是按视觉新奇度选择。

### 内容表面

用于卡片、行和长文本内容。

亮色：

```css
background: rgb(255 255 255 / 0.90);
border: 1px solid rgb(18 20 32 / 0.10);
```

深色：

```css
background: rgb(25 23 32 / 0.92);
border: 1px solid rgb(255 255 255 / 0.09);
```

模糊可选，并应保持较轻。

### 常规功能玻璃

用于 Sidebar、移动导航、Sticky Toolbar 和紧凑控件。

```css
background: rgb(255 255 255 / 0.76);
backdrop-filter: blur(20px) saturate(150%);
border: 1px solid rgb(255 255 255 / 0.64);
```

表面越大、壁纸越复杂，不透明度应越高。

### 强功能玻璃

用于 Popover、文字较多的工具栏和悬浮面板。

```css
background: rgb(250 250 255 / 0.88);
backdrop-filter: blur(24px) saturate(145%);
```

### 高密度模态玻璃

用于 Dialog、Sheet、完整简介阅读和壁纸上方菜单。

```css
background: rgb(252 252 255 / 0.94);
border: 1px solid rgb(255 255 255 / 0.60);
backdrop-filter: blur(32px) saturate(140%);
box-shadow:
  inset 0 1px rgb(255 255 255 / 0.60),
  0 28px 80px rgb(22 18 46 / 0.22);
```

深色：

```css
background: rgb(31 27 42 / 0.94);
border-color: rgb(255 255 255 / 0.10);
```

### 玻璃规则

- 表面越大、文字越多，不透明度越高。
- 不要在没有中性分隔的情况下叠加多层透明表面。
- 人物面部、服装线条或壁纸文字不能清晰穿过阅读面板。
- 不要给多个相邻玻璃控件同时添加实体 Accent 背景。
- 玻璃用于功能层级；重复内容使用实体或高不透明表面。

## 色彩系统

### 语义职责

- **Iris Indigo**：选择、当前位置、键盘焦点、主要操作。
- **Periwinkle Ambient**：环境光与装饰连续性。
- **Emerald**：已观看、完成、成功。
- **Amber**：警告、放送提醒、可恢复风险。
- **Coral Red**：破坏性操作、失败、不可逆风险。
- **Graphite**：文字、结构、元数据、中性控件。

绿色不能用于主导航。紫色不能用于所有 Hover。

### 亮色 Token

```css
--surface-solid: #ffffff;
--surface-glass: rgb(255 255 255 / 0.68);
--surface-glass-strong: rgb(255 255 255 / 0.86);
--surface-panel: rgb(250 250 253 / 0.82);
--surface-card: #ffffff;

--text-primary: #17171d;
--text-secondary: #686875;
--text-tertiary: #9595a1;

--accent-solid: #6657e8;
--accent-hover: #7567f0;
--accent-pressed: #5546c8;

--accent-foreground: #ffffff;

--accent-glow: rgb(102 87 232 / 0.26);

--watched: #16a36a;

```

### 深色 Token

```css

```

### 壁纸上方色彩规则

- UI 使用语义 Token，不使用壁纸采样色。
- `--accent-ambient` 仅用于氛围光，不用于正文。
- `--text-tertiary` 只能出现在足够不透明的表面上。
- 已观看绿色需要搭配勾、标签、线条或形状。
- 警告与破坏色需要搭配文字或图标。
- 对比度验证必须基于最终合成结果，包括壁纸、遮罩、玻璃和前景。

## 字体排版

使用系统无衬线字体栈。字体必须在玻璃表面和不同视口尺寸上保持可读。

### 共享规则

- 优先使用 Regular、Medium、Semibold 和 Bold。
- 小字号避免纤细字重。
- 正文一般为桌面 `14px` 到 `16px`，移动 `15px` 到 `17px`。
- 长文本行高为 `1.65` 到 `1.8`。
- 元数据是次要信息，不能独自承担关键含义。
- 受限 Hero/卡片中的长标题应安全换行，通常限制为两行。
- 文字放大时，将行内元数据改为堆叠，避免裁切或重叠。

### 桌面字号

- 页面标题：`30px` 到 `40px`。
- 详情 Hero 标题：`clamp(30px, 3.2vw, 52px)`。
- 区块标题：`20px` 到 `24px`。
- 卡片标题：`14px` 到 `16px`。
- 元数据：`12px` 到 `13px`。

### 移动字号

- 页面标题：`28px` 到 `34px`。
- 详情标题：`28px` 到 `40px`。
- 区块标题：`20px` 到 `24px`。
- 卡片标题：`15px` 到 `17px`。
- 元数据：`12px` 到 `14px`。

这些是范围，不是固定值。所有文字缩放下都应保持角色层级。

## 跨平台策略

桌面端与移动端共享设计语言，但可以使用不同的信息架构与组件。

### 跨平台共享

- 语义色彩与状态含义；
- 材质层级；
- 图标体系与文案语气；
- 任务结果与 API 行为；
- 加载、成功、错误与回滚语义；
- 可访问性要求。

### 桌面体验

- 有意识地使用横向空间。
- 支持可调整窗口、键盘、鼠标和 Hover 增强。
- 使用持久 Sidebar 导航。
- 空间允许时，通过相邻列保留有用的次级上下文。
- 优先使用锚定 Popover、Dialog、Inspector 和侧边 Sheet。
- 测试全宽、半屏、三分之一宽度与紧凑窗口。

桌面宽度建议：

- App Shell：最大约 `1440px`。
- 资料库：`1280px` 到 `1440px`。
- 追番：空间允许时使用主队列和最近活动双栏。
- 搜索和统计：约 `1160px` 到 `1280px`。
- 设置：约 `1120px` 到 `1240px`。

### 移动体验

- 使用 Ani Tracker 移动顶部导航作为移动端主导航模式。
- 默认不以底部 Tab Bar 替换顶部导航。任何导航重构都必须保留产品的顶部导航识别，并证明能带来明确可用性提升。
- 尊重安全区域、浏览器工具栏、方向变化和虚拟键盘。
- 大多数场景使用单一主列，只有确实清晰时才使用紧凑双列。
- 优先当前任务，并渐进披露次要信息。
- 桌面 Popover 过于拥挤时，使用 Bottom Sheet 或全屏流程。
- 重要触控热区至少 `44x44px`。
- 不能依赖 Hover。
- 保护应用内横向手势，避免与浏览器返回和垂直滚动冲突。

移动页面边距一般为 `16px` 到 `20px`，并根据安全区域调整。

### 平台适配契约

每个重要功能都必须定义两端体验。

桌面问题：

- 额外宽度如何使用？
- 操作时保留哪些上下文？
- 键盘和鼠标如何操作？
- 窗口缩小时如何变化？

移动问题：

- 首屏主要任务和操作是什么？
- 哪些内容进入渐进披露？
- Popover 是否应变为 Bottom Sheet 或全屏流程？
- 热区、安全区域和键盘避让是否正确？
- 手势是否能与浏览器和滚动手势共存？

如果只改变列数，而操作逻辑仍然面向桌面，则响应式设计并未完成。

## 布局与导航

### 桌面 Shell

- 使用持久 Sidebar 作为主导航。
- 主内容宽度取决于页面类型。
- Sticky/悬浮控件必须拥有可读的功能表面。
- 优先隐藏三级列，不要把主要内容压缩到不可读。

### 移动 Shell

- 顶部导航是 Ani Tracker 移动设计语言的稳定组成部分，而不是临时响应式降级方案。
- 移动顶部导航承载主要目的地访问，并必须适配安全区域、保持紧凑且易于触达。
- 次级目的地和低频操作可使用菜单、Sheet 或上下文控件，避免让顶部导航过载。
- Fixed 和 Sticky 元素必须包含安全区域。
- 避免持久 UI 占用过多垂直空间。

### 页面 Header

共享角色：

- 可选 Eyebrow；
- 标题；
- 可选说明；
- 可选操作。

桌面可以根据页面目的使用居中或左对齐。移动端可以省略 Eyebrow、堆叠操作，或将低优先级操作移到 Overflow。保持信息层级，而不是保持完全相同的几何结构。

## 组件语言

### 按钮

类型：

- Primary：Iris Indigo 背景、白字。
- Secondary：中性或玻璃辅助操作。
- Outline：高不透明中性表面和边框。
- Ghost：低优先级操作。
- Destructive：仅用于破坏性操作。
- Icon：带可访问标签的方形控件。

尺寸：

- 小型桌面控件：约 `32px`。
- 默认桌面控件：约 `38px`。
- 大型主要控件：约 `42px`。
- 重要移动热区：至少 `44x44px`。
- 桌面交互区域：通常至少 `28x28px`。

移动和桌面变体可以使用不同标签、Padding、分组与位置。

### 焦点

完成态 UI 不应出现浏览器默认黑色焦点框，必须替换为清晰可见的语义焦点样式。

```css
outline: none;
border-color: rgb(102 87 232 / 0.52);
box-shadow: 0 0 0 4px rgb(102 87 232 / 0.14);
```

按钮、链接、Segmented Control、Select Trigger、菜单、设置、搜索与剧集操作都必须有可见焦点。

### Segmented Control

- 使用共享移动 Thumb。
- 选中 Thumb 使用 Iris Indigo 和白字。
- Track 使用中性功能玻璃。
- 互斥选项使用 `radiogroup`。
- 只有真正 Tab Panel 使用 `tablist`。
- 支持方向键、Home 和 End。
- 标签过长或选项可能增长时，不要强行使用 Segmented Control。

### Select 与菜单

- 菜单宽度至少等于 Trigger。
- 壁纸上方使用高密度玻璃。
- 当前项使用弱 Accent 背景与勾选图标。
- 适合时支持 Escape 和点击外部关闭。
- 圆角或裁切父容器可能截断菜单时使用 Portal。
- 移动端锚定菜单过于拥挤时，改用 Bottom Sheet 或全屏选择流程。

### 卡片

- 重复内容卡片使用实体或高不透明表面。
- 桌面 Hover 可以上移约 `2px`，并轻微缩放海报。
- Hover 不能提供唯一操作入口。
- 移动卡片可以使用专门的紧凑横向布局，不必缩小桌面海报网格。
- 长标题和元数据必须适应本地化与文字缩放。

### Chip 与元数据

只使用两种视觉层级：

- 中性元数据 Chip；
- 语义状态/操作 Chip。

Provider、类型、日期和集数不能全部看起来像选中态。“下一集”可以更强，“Provider”应更弱。

## 页面模式

### 追番表

共享：

- 受保护的滑动/拖动观看交互；
- 队列摘要；
- 最近观看按本地日期分组；
- Emerald 已观看状态；
- 可见的观看状态按钮。

桌面：

- 空间允许时使用主队列和 Sticky 最近活动列；
- 最近卡片保持紧凑；
- 队列切换使用 Segmented Control。

移动：

- 使用专门的单列流程；
- 追番、补番和最近活动可使用移动专用 Tab 或区块；
- 横向拖动时保留垂直滚动；
- 触控操作始终可见且可达。

### 动画详情

该页面是状态驱动的作品控制中心，而不是数据库记录。

优先级：

1. 身份与状态。
2. 简介摘要。
3. 进度与下一步操作。
4. 元数据。
5. 有内容时的同系列条目。
6. 剧集。

桌面 Hero：

```css
grid-template-columns: clamp(180px, 17vw, 240px) minmax(0, 1fr);
grid-template-areas:
  "poster identity"
  "poster summary"
  "progress progress"
  "metadata metadata";
```

移动 Hero：

- 重组为触控友好的单列层级；
- 状态、进度和主要操作位于低优先级元数据之前；
- 空间允许时使用较小的居中海报或紧凑海报/身份组合；
- 桌面比例造成过度滚动时，不应保留桌面比例。

简介：

- Hero 只显示固定摘要；
- 完整简介永远不展开 Hero；
- 宽桌面打开右侧阅读 Sheet；
- 窄桌面使用居中 Dialog；
- 移动使用 Bottom Sheet 或全屏阅读页。

Hero 插画：

- 插画光晕保留在海报和身份区域附近；
- 中性遮罩保护文字；
- 光晕进入进度和元数据前衰减；
- 元数据使用稳定的高不透明表面。

同系列条目：

- 过滤当前作品；
- 没有其它条目时不渲染；
- 区块只负责内容和跳转；
- 维护操作放在 Hero 设置菜单中。

### 资料库

- 搜索/筛选状态保持 URL 可寻址。
- 桌面网格可使用 `repeat(auto-fill, minmax(190px, 1fr))`。
- 卡片通过高不透明内容表面，在全局壁纸上保持可读。
- 桌面筛选可以使用锚定控件。
- 桌面 Popover 过于拥挤时，移动筛选可以使用专用 Sheet。
- 移动卡片可以使用紧凑布局，而不是缩小桌面网格。

### 搜索

- 桌面可以使用横向结果卡片和锚定 Provider 控件。
- 移动可以使用堆叠或紧凑媒体行及键盘感知控件。
- 主要添加操作使用 Iris Indigo。
- 已添加状态变为中性或成功状态，不再作为主要按钮。

### 统计

- 已观看/完成热力图使用 Emerald。
- 选择/焦点使用 Iris Indigo。
- Tooltip 使用功能玻璃。
- 桌面可并列多个图表。
- 移动一次优先一个图表，并用点击/焦点取代 Hover-only 详情。

### 设置

- 对相关设置进行分组。
- 避免每个设置独占一张巨型卡片。
- 桌面可使用分类导航和内容 Pane。
- 移动应使用分组列表、Drill-in 页面或紧凑区块，而不是压缩桌面 Split View。
- 自动保存确认应短暂出现，而不是长期重复。

## 受保护的剧集交互

剧集观看切换是受保护的产品行为，也是 Ani Tracker 剧集更新操作的核心之一。

剧集卡片横向拖拽是参考 TV Time 交互体验后做出的必要产品选择。它不是附带动画或可随意删除的视觉效果，而是快速、直接更新剧集状态的核心路径，并推荐移动端用户使用。

Ani Tracker 同时提供可见的 Checkbox 风格状态按钮。拖拽和 Checkbox 是完成同一剧集状态操作的两个一级入口：

- 推荐移动端用户通过横向拖拽剧集卡片快速更新状态。
- 用户始终可以点击 Checkbox 风格按钮设定或取消已观看状态。
- 键盘和辅助技术用户通过 Checkbox 风格按钮操作。
- 两种路径不得产生不同的数据语义、确认规则或回滚行为。

- 移动拖动阈值：`76px`。
- 桌面拖动阈值：`224px`。
- 左拖标记已观看。
- 右拖取消已观看。
- API 路径保持 `await onChange(next)`。
- 乐观更新与失败回滚保持不变。
- 移动边缘返回保护保持启用。

详细交互规则以当前代码实现为行为权威来源：

- `web/src/components/library/EpisodeWatchToggle.tsx`
- `web/src/components/library/TrackingEpisodeRow.tsx`
- `web/src/components/library/EpisodeRow.tsx`

除非有明确产品决策修改，否则必须保留当前的方向判断、轴向锁定、边缘返回保护、不可用方向处理、确认行为、Pending 状态、Checkbox 语义、API 调用和失败回滚。

视觉结构：

- 动作背景位于内容后方；
- 内容和观看按钮一起移动；
- 绿色确认已观看/完成；
- 紫色可以表达完成前的交互提示。

平台可以使用不同实现，但方向语义和数据行为必须一致。移动端应让拖拽能力容易被发现或自然学习，但不应使用阻塞式教程。Checkbox 风格按钮继续作为精确且可访问的替代入口保持可见。

## 动效

动效用于解释状态与空间关系。

时长：

- 快速反馈：`120ms` 到 `160ms`。
- 标准过渡：`180ms` 到 `220ms`。
- Sheet/Dialog 进入：`220ms` 到 `260ms`。

缓动：

```css
--ease-standard: cubic-bezier(0.2, 0, 0, 1);
--ease-emphasized: cubic-bezier(0.2, 0.8, 0.2, 1);
```

桌面动效可强调 Hover 和锚定来源。移动动效可强调直接操控、Sheet 和导航连续性。

避免：

- 持续壁纸运动；
- 默认壁纸视差；
- 过多模糊动画；
- 常规状态变化后的全页重新淡入；
- 与手势无关的强反弹；
- 延迟下一步操作的动效。

减少动态效果时，将非必要位移/缩放改为短淡入或取消动画。

## 可访问性

- 合成后的文字对比度必须考虑壁纸、遮罩、玻璃和前景。
- 18pt 以下文字目标对比度至少 `4.5:1`。
- 重要移动热区至少 `44x44px`。
- 桌面控件通常至少 `28x28px`。
- 状态除颜色外还需要文字、形状或图标。
- 键盘导航不能依赖 Hover。
- Dialog 和 Sheet 需要明确标签、关闭操作和焦点处理。
- 适合时，Escape 关闭桌面瞬态 Overlay。
- 移动 Overlay 尊重安全区域和虚拟键盘。
- 文字缩放不能重叠控件或隐藏关键信息。
- 减少透明度时使用实体表面。
- 减少动态效果时移除非必要动效。
- 核心手势始终存在可见替代操作。

## 实现规则

### 语义 Token 优先

颜色、表面、边框、文字、状态、阴影、圆角与动效使用语义 Token。避免组件局部颜色重复 Token 含义。

### Portal 规则

以下情况通过 `document.body` 渲染：

- Overlay 位于 `overflow: hidden` 或裁切父级内；
- 需要越过圆角 Hero/卡片；
- 需要在插画和元数据上方保持稳定 z-index；
- 需要视口碰撞处理。

桌面和移动可以共享 Overlay 内容，同时使用不同 Shell。

### 圆角层级

- 小型/默认控件：`10px` 到 `12px`。
- Input：约 `16px`。
- 内容卡片：约 `18px`。
- Panel：约 `24px`。
- Dialog/Sheet：约 `28px`。
- Pill：仅用于 Segmented Control、Tag 和明确胶囊语义。

### 阴影层级

```css
--shadow-low:
  0 1px 2px rgb(18 20 32 / 0.05),
  0 8px 24px rgb(18 20 32 / 0.06);

--shadow-medium:
  0 2px 6px rgb(18 20 32 / 0.06),
  0 16px 40px rgb(18 20 32 / 0.10);

--shadow-high:
  0 8px 24px rgb(18 20 32 / 0.12),
  0 34px 90px rgb(18 20 32 / 0.18);
```

深色模式更多依赖表面明度、克制边框和一像素内部高光，而不是大面积暗色阴影。

每个重要视觉改动都必须检查以下场景。

### 壁纸场景

- 明亮高调插画；
- 深色低调插画；
- 高饱和多色插画；
- 导航后方存在密集人物细节；
- 文字后方存在强黑白边缘；
- 默认 Twilight Iris 回退背景。

### 外观场景

- 亮色模式；
- 深色模式；
- 减少透明度；
- 减少动态效果；
- 增大浏览器文字缩放。

### 平台场景

- 宽桌面；
- 半宽桌面窗口；
- 紧凑桌面/类似平板宽度；
- 窄移动竖屏；
- 支持时的移动横屏；
- 触控和鼠标输入；

- 超长标题；
- 超长简介；
- 缺少海报；
- 缺少元数据；
- 空同系列条目；
- 大量剧集；
- 加载、错误和乐观更新回滚状态。

## 应该与不应该

### 应该

- 将用户插画视为基础环境约束。
- 在保护任务内容的同时保留壁纸可见性。
- 使用玻璃区分功能层、插画和内容。
- 随文字密度和表面尺寸提高不透明度。
- 使用 Iris Indigo 表达选择、焦点和主要操作。
- 使用 Emerald 表达已观看/完成。
- 必要时分别设计桌面和移动流程。
- 使用符合平台习惯的 Overlay 和导航。
- 为手势与 Hover 提供替代操作。
- 在极端壁纸上测试最终合成结果。

### 不应该

- 不在原始壁纸上直接放置正文。
- 不只依赖模糊解决对比度。
- 不为所有重复内容卡片使用重玻璃。
- 不让壁纸细节清晰穿过阅读表面。
- 不根据每张壁纸改变产品 Accent。
- 不用绿色表达常规选择。
- 不把紫色用于所有 Hover 或控件背景。
- 不把移动端当作压缩桌面布局。
- 不把桌面端当作拉宽移动布局。
- 不把关键操作隐藏在 Hover 后。
- 未经明确产品批准，不修改受保护的观看滑动语义。

## 产品身份

Ani Tracker 是一个专注的媒体效率工具。它允许用户喜爱的动画插画成为体验的一部分，同时不牺牲清晰度。

该设计语言不是通用 Glassmorphism，而是一个自适应分层系统：

- 用户插画提供身份；
- 自适应环境层控制视觉噪声；
- 内容表面保护阅读；
- 功能玻璃建立控件层级；
- Iris Indigo 表达方向和操作；
- Emerald 确认完成；
- 平台专用结构让桌面和移动都显得经过专门设计。

未来设计必须保持这一层级、持续遵循 Apple 设计原则，并分别通过桌面和移动体验评估后，才可视为完成。
